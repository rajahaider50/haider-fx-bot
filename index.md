## 1. Project Overview

This document defines the complete architectural blueprint for a **consent-based family device management application**. The system is built around a **transparent pairing model**: a guardian account links to one or more family member devices only after an explicit, visible pairing step (QR code or pairing code) is completed on the device being registered. The app is designed to run openly and visibly on every enrolled device — it is not designed to be hidden, disguised, or installed without the device holder's knowledge, and the blueprint assumes this constraint throughout (e.g., the app respects the Android OS's built-in camera/microphone privacy indicators and does not attempt to suppress them).

The platform consists of:
- An **Android client app** installed on both guardian and family-member devices (single codebase, role-driven UI).
- A **backend API** for authentication, device registry, sync, and command relay.
- A **cloud data layer** for backup, analytics, and cross-device state.

## 2. Vision & Goals

**Vision:** Give families a transparent, affordable, and reliable way to stay connected, coordinate safety, and manage screen habits — without requiring paid enterprise tooling.

**Goals:**
- Zero-cost operation for small deployments (5–20 devices) using free-tier infrastructure.
- Fully transparent operation — every monitored device shows the app is installed and active.
- Modular, phase-based delivery so the product is always in a working, installable state.
- Production-grade reliability: stable background services, resilient sync, auditable logs.
- Clear separation of guardian privileges vs. family-member privileges (RBAC).

## 3. Project Scope

**In scope:**
- Location sharing, geofencing, and arrival/departure alerts (with device holder's knowledge that sharing is active).
- Screen time, app usage tracking, and app time-limit enforcement.
- Device health monitoring (battery, storage, connectivity).
- Remote device conveniences (ring, lock, flashlight, volume) initiated by the guardian and visible to the device holder.
- Family communication (check-ins, SOS, broadcasts).
- Reporting, analytics, backup/restore, and multi-device sync.

**Out of scope / explicitly excluded:**
- Covert or hidden operation (no disguised app icons, no suppression of OS privacy indicators, no silent installation).
- Interception of a device holder's private messages, calls, or third-party app content.
- Any feature whose primary purpose is to monitor a person without their knowledge that the app is present and active on their device.

> **Design Note:** Several sensitive capabilities appear in typical "family safety" feature catalogues (e.g., camera/microphone access, screen mirroring). Wherever your pasted feature list includes these, the corresponding module sections below describe them as **guardian-initiated, on-device-notified actions** (the family member sees an active indicator and a log entry), consistent with the transparency principle above and with Android's platform privacy protections, which cannot and should not be bypassed.

## 4. User Roles

| Role | Description | Key Privileges |
|---|---|---|
| **Super Admin** | Platform owner/operator | Full system configuration, all family accounts (if hosting multiple families) |
| **Guardian/Parent** | Primary account holder for a family | Manage devices, view reports, configure automation, receive alerts |
| **Co-Guardian** | Secondary parent/guardian | Same as Guardian, scoped to shared family group |
| **Family Member/Child** | Device holder being coordinated with | Views own status, receives visible notifications, can check in, can send SOS |
| **Guest/Viewer** (optional) | Read-only relative (e.g., grandparent) | View-only dashboard access, no control actions |


## 5. System Architecture

```
┌─────────────────────┐        ┌──────────────────────┐        ┌───────────────────┐
│  Guardian Android    │◄──────►│   Backend API Layer  │◄──────►│  Cloud Data Store  │
│  App (Admin Role)    │  HTTPS │  (Auth, Sync, Relay)  │        │  (DB + Storage)    │
└─────────────────────┘        └──────────────────────┘        └───────────────────┘
          ▲                              ▲
          │  Push / Realtime             │  Push / Realtime
          ▼                              ▼
┌─────────────────────┐        ┌──────────────────────┐
│ Family Member Device │◄──────►│ Background Service    │
│ (Client Role)        │        │ Layer (on-device)     │
└─────────────────────┘        └──────────────────────┘
```

- **Client tier:** Single Android codebase, role-based UI (Guardian vs. Family Member), offline-first with sync queue.
- **API tier:** Stateless REST API + realtime channel (WebSocket or Firebase Realtime DB) for command relay and live status.
- **Data tier:** Cloud database (structured) + object storage (media/backups) + local encrypted database on-device.
- **Command relay pattern:** Guardian actions (e.g., "ring device") are queued as signed commands, delivered via push, executed by the on-device background service, acknowledged back through the API, and always surfaced in an on-device activity/notification log visible to the family member.

## 10. Recommended Android Application Architecture

- **Pattern:** Clean Architecture (Data / Domain / Presentation layers) + MVVM within the Presentation layer.
- **UI Toolkit:** Jetpack Compose (modern, less boilerplate) with Material 3.
- **DI:** Hilt.
- **Async:** Kotlin Coroutines + Flow.
- **Navigation:** Jetpack Navigation Compose with a single-activity architecture.
- **Modularization (as the app grows):** `:core`, `:data`, `:domain`, `:feature-devices`, `:feature-reports`, etc., to keep build times manageable and enforce boundaries.

## 11. Database Architecture

**On-device (local):** Room (SQLite) with SQLCipher for at-rest encryption of sensitive tables (location history, credentials cache).

**Cloud (server-side):** A relational database is recommended for this domain (structured family/device/rule relationships benefit from relational integrity) — see Free-First Stack for exact free-tier providers.

**Core Entities (indicative, not exhaustive):**
`users`, `family_groups`, `devices`, `device_health_snapshots`, `location_points`, `geofences`, `app_usage_events`, `automation_rules`, `rule_executions`, `commands`, `audit_logs`, `notifications`, `backups`.

**Design Principles:** Every table has `created_at`/`updated_at`; soft-delete (`deleted_at`) instead of hard delete for auditability; foreign keys scoped to `family_group_id` for tenant isolation.

## 12. API Architecture

- **Style:** REST over HTTPS, JSON payloads; versioned via `/api/v1/...`.
- **Auth:** Bearer JWT access tokens (short-lived, ~15 min) + refresh tokens (rotating, revocable).
- **Resource-oriented endpoints**, e.g. `/api/v1/devices`, `/api/v1/devices/{id}/commands`.
- **Realtime channel:** WebSocket (or a managed realtime DB) for command delivery and live status, with REST polling as a documented fallback.
- **Rate limiting:** Per-account and per-IP limits on auth endpoints to mitigate brute force.
- **Error format:** Consistent envelope `{ "error": { "code": "...", "message": "..." } }`.

## 13. Cloud Architecture

Recommended shape: **Backend-as-a-Service (BaaS) first** (e.g., Firebase or Supabase) to minimize operational overhead for a small family deployment, with a clear migration path to a self-hosted API if the project scales. See Section 34 for specific free-tier providers.

## 14. Sync Architecture

- **Model:** Offline-first. All writes go to the local DB first, then queued for sync.
- **Conflict resolution:** Last-write-wins keyed by server timestamp for most entities; append-only/event-sourced pattern for logs and location history (no conflicts possible since these are insert-only).
- **Delivery guarantee:** At-least-once with idempotency keys on the server to safely handle retries/duplicates.

## 15. Security Architecture

Encryption in transit (TLS 1.2+, certificate pinning), encryption at rest (AES-256 for local DB, provider-managed encryption in the cloud), JWT-based auth with rotating refresh tokens, RBAC enforced server-side on every request, comprehensive audit logging, and rate-limited/lockout-protected authentication. Security-sensitive actions (adding a device, exporting data, changing a password) require re-authentication or 2FA where enabled.

## 16. Automation Architecture

Trigger → Condition → Action pipeline evaluated by the on-device Background Service Engine for device-local triggers (screen on/off, battery) and by the server for time/schedule-based triggers, keeping cloud-side evaluation stateless and idempotent.

## 17. Background Service Architecture

Foreground Service (for continuous, user-visible monitoring like active location sharing) + WorkManager (for deferrable periodic sync/health checks) + a lightweight watchdog that restarts services after OS kill events, always respecting Android's foreground-service notification requirement so the family member is never unaware that a service is active.


## 18. Device Management Architecture

Central `devices` registry with public-key binding created at pairing time; every remote command is signed by the guardian's authenticated session and verified on-device before execution; device removal immediately revokes its keys and pending commands.

## 19. Analytics Architecture

Event-based ingestion (`usage_events`, `location_points`) → scheduled server-side rollup jobs (hourly/daily) → pre-aggregated `daily_rollups` table for fast report queries → client charts read only from rollups, never raw event tables, to keep the UI responsive.

## 20. Permission Architecture

Every Android runtime permission is requested **contextually** (at the point the corresponding feature is enabled) with an in-app rationale screen shown first. A single "Permissions" screen in Settings lists every permission the app holds, why it's needed, and a one-tap path to the system settings screen to revoke it. No permission is silently re-requested after a user denial without a fresh, visible rationale.

## 21. Notification Architecture

Categorized notification channels (Android `NotificationChannel`) for: Alerts (SOS, geofence), Status (sync, low battery), Guardian Actions (remote command executed), and Digest (weekly report). Users can mute non-critical channels independently; SOS/emergency channel cannot be muted.

## 22. Error Handling Architecture

Centralized error-mapping layer translating network/DB/business errors into user-facing messages; global error screens for No Internet / Server Error / Access Denied; all errors logged with correlation IDs for support diagnosis; graceful degradation (e.g., show cached data with a "stale" indicator when offline).

## 23. Logging Architecture

Structured JSON logs client-side (rotated, size-capped, encrypted at rest) and server-side (centralized log aggregation). Log levels: DEBUG (dev only) / INFO / WARN / ERROR / SECURITY. Security-relevant logs (auth, permission changes, command execution) are append-only and retained longer than general activity logs.

## 24. Backup & Restore Architecture

Scheduled encrypted backups of settings, rules, and family/device metadata to cloud storage; manual "Backup Now"; restore flow validates a checksum and shows a preview/diff before applying; local export/import supported for users who prefer not to use cloud backup.

## 25. Update & Version Management

Semantic versioning (`MAJOR.MINOR.PATCH`). In-app update-check against a `/app/version` endpoint; soft-prompt for minor updates, hard-block (Play Store "in-app update" API, blocking flow) only for updates that fix a security issue. Full changelog surfaced in-app.

## 26. License System

For a free/self-hosted small-family deployment, no license server is required. If offering a hosted/paid tier later, recommend a simple entitlement flag on the `family_groups` record (`plan: free|plus`) checked server-side, rather than client-side license keys (which are trivially bypassed).

## 27. Device Verification & Pairing Flow

1. Guardian taps "Add Device" → server issues a short-lived pairing token (QR + 6-digit fallback code, 10-minute expiry).
2. Family member opens app → "Join Family" → scans QR or enters code.
3. Family member's device generates a keypair; public key + device metadata sent to server with the pairing token.
4. Server validates token, registers device, notifies guardian's app in real time.
5. Guardian sees "New device paired" confirmation and can rename/configure it.

Device verification for subsequent sessions relies on the registered device key signing each request/command acknowledgment.

## 28. UI/UX Guidelines

**Color System:** Primary (brand), Secondary (accent), Semantic colors — Success (green), Warning (amber), Danger (red), Info (blue) — plus Surface/Background pairs for light and dark themes, all meeting WCAG AA contrast.

**Typography:** A single type scale (Display/Headline/Title/Body/Label per Material 3 spec), one primary font family, consistent line-heights.

**Icons:** One icon set (Material Symbols) throughout; consistent stroke weight; avoid mixing icon styles.

**Components:** Reusable card, button (filled/outlined/text), chip, badge, and list-item components defined once in a shared design-system module.

**Cards:** Consistent corner radius (12–16dp), elevation level 1 for resting state, level 2 on interaction.

**Dialogs:** Reserved for irreversible or high-stakes confirmations only (e.g., remove device, delete data); everything else uses snackbars/bottom sheets to avoid dialog fatigue.

**Bottom Sheets:** Used for contextual actions and multi-step pickers (e.g., choosing an automation trigger).

**Navigation:** Bottom nav (≤5 top-level destinations) + drawer for secondary items; consistent back-stack behavior.

**Animations:** Motion duration 150–300ms, standard easing curve; used to reinforce hierarchy (shared-element transitions for drill-in), never purely decorative.

**Loading States:** Skeleton placeholders matching final content shape; avoid indeterminate spinners for anything longer than ~1 second.

**Empty States:** Friendly illustration + one-line explanation + primary action (e.g., "No devices yet — Add your first device").

**Success/Error States:** Success via snackbar/checkmark animation; errors via inline messaging first, full-screen error state only for unrecoverable/global failures.

## 29. Accessibility Guidelines

Minimum 4.5:1 text contrast; all touch targets ≥48dp; full screen-reader labeling (`contentDescription`) on every interactive element; support system font-scaling up to 200% without layout breakage; respect system "reduce motion" setting; color never used as the sole indicator of state (pair with icon/text).

## 30. Performance & Battery Optimization Guidelines

Batch network calls, use WorkManager's battery/network constraints for non-urgent syncs, prefer geofencing APIs over continuous GPS polling, request battery-optimization exemption only for the minimum necessary foreground service, profile with Android Studio's Profiler each phase, target cold start <2s on mid-range devices, and cap background CPU wake-ups via coalesced alarms.

## 31. Privacy Guidelines & Security Best Practices

Collect the minimum data necessary for each enabled feature; display a clear, plain-language privacy notice at pairing time explaining exactly what will be visible to the guardian and that the app runs openly (visible icon, foreground-service notification) on the family member's device; provide in-app data export and deletion; never request a permission for a feature that is disabled; store secrets using the Android Keystore, never in plain SharedPreferences.

## 32. Development Standards & Naming Conventions

**Coding Standards:** Kotlin official style guide, ktlint/detekt enforced in CI, unit tests required for all use-case/business-logic classes.

**Naming Conventions:** PascalCase for classes/composables, camelCase for functions/variables, SCREAMING_SNAKE_CASE for constants.

**File Naming:** Feature-scoped folders (`devices/DeviceListScreen.kt`, `devices/DeviceViewModel.kt`).

**Database Naming:** snake_case table/column names, singular entity class names mapped to plural table names (`Device` → `devices`).

**API Naming:** Plural nouns for collections (`/devices`), nested resources for relationships (`/devices/{id}/commands`), verbs avoided in favor of HTTP methods.

## 33. Testing / QA / Deployment / Maintenance Checklists

**Testing Checklist:** Unit tests for domain/use-case layer, instrumented tests for DAOs, UI tests for critical flows (pairing, login), manual test pass on min and max supported Android versions.

**QA Checklist:** Feature acceptance criteria met, no crashes in a 30-minute soak test, permissions flow verified on a fresh install, regression suite green.

**Deployment Checklist:** Version bump, changelog updated, signed release build, Play Store internal-testing track validated before production rollout.

**Maintenance Checklist:** Dependency updates reviewed monthly, crash-report dashboard checked weekly, security advisories monitored, database backup restore drill quarterly.

## 36. GitHub Workflow & CI/CD

**Branching Model:** `main` (always releasable) ← `develop` (integration) ← `feature/*` branches, merged via Pull Request with required review and passing CI.

**Recommended GitHub Actions Pipeline (`.github/workflows/android-ci.yml`):**
1. **Lint & Static Analysis** — ktlint, detekt, Android Lint.
2. **Dependency Check** — Gradle dependency resolution, flag outdated/vulnerable dependencies.
3. **Unit Tests** — run JVM unit tests, publish coverage report.
4. **Instrumented Tests** — run on an emulator via `reactivecircus/android-emulator-runner`.
5. **Build** — assemble debug APK on every PR; assemble signed release APK on `main` merges/tags.
6. **Artifact Upload** — upload the APK as a GitHub Actions build artifact for manual testing/download.
7. **Release** — on a version tag, create a GitHub Release with the signed APK attached and auto-generated changelog.

**Issue Tracking:** GitHub Issues with labels (`bug`, `feature`, `phase-N`, `blocked`) and a GitHub Project board with columns matching the 20 phases.

**Error Investigation Policy:** Any CI failure or reported bug is triaged to identify root cause (not just symptom) before a fix is merged; the phase does not advance while an unresolved failure remains open.
